# 今日やる / 今後やる / カレンダー 設計メモ

## 方針

今後は「今日かどうか」を専用フラグで持つのではなく、タスクに予定日を持たせる。

予定日から、今日やるもの・今後やるもの・カレンダーに出すものを判定する。

## 基本のタブ

- 今日やる
- 今後やる
- カレンダー

## 共通UI・方針

- ログアウトは全画面共通のフッターとして最下部に表示する。
- タグはチップ表示のみ。タグでの絞り込み機能は持たない（個人利用ではオーバースペックのため）。

## データ設計

タスクには予定日を持たせる。

```ts
Item {
  id: string
  title: string
  tag: string | null
  recurring: boolean
  scheduledDate: string | null // YYYY-MM-DD。未定なら null
  status: "open" | "done"
  createdAt: string
  doneAt?: string
}
```

現状の `today_items` は `scheduledDate` に修正する。

毎日タスク（`recurring = true`）は `scheduledDate` を持たない（null）。
毎日タスクの完了は、タスク本体の `status` ではなく、**日ごとのできたことログで判定する**
（`status = "done"` にすると恒久的に完了扱いになり事故るため）。

実装上の注意：`completeItem()` は `recurring = true` のとき**絶対に `status = "done"` にしない**（ガードを入れる）。
型上は `status` が毎日タスクにも存在してしまうので、将来は型を分けてもよい（今はガードで十分）。

## 表示条件

### 今日やる

以下を表示する。

- 毎日タスク
- `scheduledDate` が今日以前の未完了タスク
  （今日やる画面で新規追加したタスクは `scheduledDate = 今日` になるので、自動的にここに含まれる）
- 今日完了したタスク（打ち消し線で残す）

昨日以前に予定していて未完了のタスクも、今日やるに残る。
完了して残すのは「今日完了したもの」に限定する（先週予定して先週完了したものは出さない）。

### 今後やる

以下を表示する。

- 日付未定の未完了タスク
- `scheduledDate` が明日以降の未完了タスク

毎日タスクは表示しなくてよい。

### カレンダー

以下を表示する。

- 予定日が入っている未完了タスク
- できたことログ
- その日の「ひとこと」

繰り越し中の未完了タスク（`scheduledDate` が過去）も、その予定日（過去日）に表示してよい。
そのため、同じ繰り越しタスクが「今日やる」とカレンダーの過去日の両方に出ることがある（許容する）。

## 今日やる画面

今日やる画面は実行と編集の両方ができる。

- 今日の日付の新しいタスクを追加できる
- 今日の既存タスクを編集できる
- 今日の既存タスクを削除できる
- 毎日タスクも表示する
- 毎日タスクも編集できる
- チェックボックスを表示する
- チェックすると完了状態になる
- 完了すると「できたこと」として今日の日付で記録する
- 今日完了したタスクは今日やる一覧に残す（打ち消し線）
- 完了状態は取り消せる
- 小さなステップを追加・削除・完了できる
- 一度きりのタスクを、今日やるから外せる（`scheduledDate = null` にして今後やるへ戻す。毎日タスクは対象外）

既存の `/36達成` のような達成数表示は維持する。

## 今後やる画面

今後やる画面は予定整理と編集の場所。

- 日付未定の新しいタスクを追加できる
- 日付指定した新しいタスクを追加できる
- 既存タスクを編集できる
- 既存タスクを削除できる
- 日付を変更できる
- 日付なしにできる
- 今日やるにできる
- 毎日タスクは表示しなくてよい
- チェックボックスは表示しない
- 完了操作はしない

今後やる画面でチェック完了はしない。
未来予定のタスクを今日やったことにしたい場合は、まず今日やるにしてから、今日やる画面でチェックする。

## 今日やる画面と今後やる画面の共通化

今日やる画面と今後やる画面は、できるだけ同じUI・同じDOM・同じ関数で実装する。

共通にするもの:

- タスク行
- タスク追加フォーム
- 編集
- 削除
- タグ表示
- 日付変更
- 小さなステップの表示・編集

モードごとに変えるもの:

```tsx
<TaskListView mode="today" />
<TaskListView mode="future" />
```

today モード:

- 今日やる条件で絞り込む
- 毎日タスクを表示する
- チェックボックスを表示する
- 完了ログを作る
- 追加時の `scheduledDate` は今日

future モード:

- 今後やる条件で絞り込む
- 毎日タスクは表示しない
- チェックボックスは表示しない
- 完了ログは作らない
- 追加時に日付未定または日付指定を選べる

## カレンダー画面

カレンダーは、予定と記録を閲覧する場所。

日付を押すと、その日の情報を表示する。

表示するもの:

- その日に予定されている未完了タスク
- その日にできたタスク
- その日の「ひとこと」

カレンダー上の `+` から、その日に予定されたタスクを追加できてもよい。
この場合、追加されるタスクの `scheduledDate` は選択中の日付になる。

## ひとこと

日ごとの文章は「メモ」ではなく「ひとこと」と呼ぶ。

理由:

- タスクと混同しにくい
- カレンダーの日付に紐づく短い記録として扱いやすい

既存の `day_notes` は、UI上では「ひとこと」と表示する。

## 完了ログ

完了ログは、実際にチェックした日付で記録する。

```txt
doneLog.date = チェックした日
scheduledDate = 予定日
```

「今日完了したか」の判定は、`item.status` や `doneAt` ではなく
**できたことログ（その日付の doneLog があるか）** で行う。
これで毎日タスクも一度きりタスクも同じ基準で判定できる。

参考：今日やるに出す集合は、
`毎日タスク ∪ (scheduledDate ≤ 今日 かつ status=open) ∪ (今日の doneLog があるもの)`
となる（最後の集合が「今日完了して打ち消し線で残す」分）。

完了操作は今日やる画面でのみ行う。

## 保存失敗時の扱い

書き込み（Supabase への保存）が失敗したときは、最小限の通知だけ出す。

- 失敗時にトーストを表示する（例：「保存に失敗しました。通信を確認して、もう一度お試しください」）。
- 主な想定原因は通信断（オフライン・電波が悪い）。ただし保存失敗の理由は限定しない（認証切れ・将来の migration 漏れ等もありうる）。
- フルの再同期やリトライ機構までは作らない。まずは「失敗したと気づける」ことを優先する。
- 注意：楽観更新のままなので、トーストだけだと画面上は保存成功に見えることがある。不安が出てきたら「失敗時に再読み込みする／再読み込みボタンを出す」程度を検討する（今は不要）。
